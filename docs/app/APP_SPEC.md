## Network Layer Principles

Binding on all future briefs/instances touching the Danger Map network layer (report collection, warnings distribution, telemetry) — the reporting/watching client and the receiving service alike.

a. **Consent as architecture** — nothing leaves the machine without an explicit human tap, ever. No background auto-submission, no "send by default," no dark patterns that make declining harder than accepting.
b. **Pattern not payload** — reports carry `threat_class`, `indicator`, `platform`, `profile_version` only; never prompts, paths, file contents, or user context. The outward-facing warnings feed enforces this at the read boundary regardless of what the write boundary currently accepts.
c. **Corroboration before escalation** — a threat escalates on multiple independent observed reports, never one.
d. **Observed vs suspected report grades are distinct end to end** — the `grade` field is never inferred, defaulted away, or blurred between collection, storage, and the warnings feed.
e. **C-Cat proposes, the human disposes** — no silent profile rewrites, no auto-submission. Every report and every clean bill submission is something the operator explicitly saw and approved before it left the machine.
f. **Clean Bills are versioned telemetry** — wall x platform_version x held/failed enables fleet-level platform-drift detection: a wall that quietly stops holding across many machines on the same platform version is a signal, not noise.

## Assignment Model

Decision log, 15 Jul 2026. Binding on all future briefs/instances touching discovery, apply, prove, tray, vet, or any UI surface (Slider, Feed, tray icon, CLI) that states or implies a protection claim.

a. **A compiled profile protects nothing until applied to a target.** `compile` writes profile files to curiosity-cat's own home directory (`profiles/`) — it does not touch any target's real configuration. Nothing is guarded, nothing is enforced, and no UI text may imply otherwise, until an explicit `apply` step installs that profile into a specific target's real settings.
b. **Targets are discovered, not assumed.** The estate — every Claude Code project, the global `~/.claude` settings, every running agent process, every configured MCP server — is enumerated by discovery, never inferred from what happens to be open or recently used. Protection state is per-target, not global: one project having an applied profile says nothing about any other target's state. Per-target protection state is always visible, never folded into a single ambiguous app-wide "protected" flag.
c. **Apply installs into the target's real settings, with backup and undo.** Applying a profile writes into the target's actual configuration (a project's `.claude/settings.json`, the global `~/.claude/settings.json`, etc.), first backing up whatever was there so the operator can always undo back to the pre-apply state. Apply is the only step that is allowed to touch a target's real settings; compile, prove, and discovery never do.
d. **Prove runs against applied reality.** Once a profile is applied to a target, `prove` exercises trials against that target's actual installed configuration — not a copy, not the profile directory in isolation. A Clean Bill is a claim about what is currently enforced at that target, so it has to observe the same file the target's own tools read.
e. **All UI text must answer: what is protected, from what, since when.** Every surface that shows a protection claim — Slider, Feed, tray icon, Clean Bill viewer, CLI output — states the level applied (what), the walls that level holds (from what), and the apply timestamp (since when). A bare "protected" or "guarded" with no level, no wall list, and no date is not an honest claim under this model and may not ship.
f. **The tray icon reflects the worst protection state across all targets.** With multiple targets in play, the tray shows the least-protected one, never an average, a majority, or the best case. An operator glancing at the tray icon sees the estate's actual weakest point, not a flattering summary.
g. **The compile result view must state plainly when a profile is not yet assigned.** Because of (a), every compile result — CLI output, Slider confirmation, wherever a freshly compiled profile is first shown — says outright that this profile is not yet applied to any target and protects nothing until it is, rather than leaving that unstated and letting the operator assume otherwise.
